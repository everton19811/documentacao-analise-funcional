# 📚 4. Glossário de Análise Funcional

| Termo | Definição |
| :--- | :--- |
| **AF** | Analista Funcional |
| **BA** | Business Analyst (Analista de Negócio) |
| **BABOK** | Business Analysis Body of Knowledge — guia do IIBA |
| **BPMN** | Business Process Model and Notation — notação padrão para modelar processos |
| **BRD** | Business Requirements Document — documento de requisitos de negócio |
| **CR** | Change Request — solicitação formal de mudança |
| **DoR** | Definition of Ready — critérios para uma US entrar em sprint |
| **DoD** | Definition of Done — critérios para uma US ser considerada concluída |
| **Elicitação** | Ato de extrair/descobrir requisitos junto aos stakeholders |
| **Épico** | Grande bloco de trabalho, geralmente dividido em várias features |
| **Feature** | Funcionalidade que agrega valor, composta por várias user stories |
| **FRD** | Functional Requirements Document — o que o sistema deve fazer |
| **GAP Analysis** | Análise comparativa entre estado atual (As-Is) e estado desejado (To-Be) |
| **Gherkin** | Sintaxe Given/When/Then para descrever cenários de aceite |
| **IIBA** | International Institute of Business Analysis |
| **INVEST** | Independent, Negotiable, Valuable, Estimable, Small, Testable — qualidades de uma US |
| **Key user** | Usuário-chave da área de negócio, referência para requisitos e UAT |
| **MoSCoW** | Must / Should / Could / Won't — técnica de priorização |
| **NFR** | Non-Functional Requirement — requisito não-funcional (performance, segurança) |
| **PO** | Product Owner |
| **RN** | Regra de Negócio |
| **RTM** | Requirements Traceability Matrix — matriz de rastreabilidade |
| **SIPOC** | Suppliers, Inputs, Process, Outputs, Customers — visão macro de processo |
| **Sign-off** | Aceite formal do documento/entrega pelo stakeholder |
| **SRS** | Software Requirements Specification (IEEE 830) |
| **Stakeholder** | Qualquer parte interessada no projeto |
| **STAR** | Situation, Task, Action, Result — técnica para descrever conquistas |
| **UAT** | User Acceptance Test — homologação pelo usuário |
| **UC** | Use Case — caso de uso (UML) |
| **US** | User Story |
| **Walkthrough** | Revisão passo a passo do documento com stakeholders |
| **5W2H** | What, Why, Where, When, Who, How, How much — checklist de elicitação |
